# Yearly Review/Plan 流程

每年一次。模板只有 frontmatter（`theme` / `vision`），正文几乎空白 — **这一层主要靠对话**，输出是用户的语言而非数据表格。

## Mode + 触发时机

入参 `/obsidian-review yearly [<mode>]`，mode 缺省 = Review + Plan 连贯。

| Mode | 走什么 |
|---|---|
| 缺省 | **Review 上年 + Plan 本年** 连贯（推荐年初前 2 周，可分 2-3 次完成） |
| `review` | 只 Review 上年 |
| `plan` | 只 Plan 本年（仍读上年 review 数据作为输入） |

Review 段额外数据源：
- 本年 4 个 quarterly 的全部 frontmatter + 「💡 季度关键洞察」+ 「📈 季度对比分析」
- 本年 12 个 monthly 的 frontmatter（theme 序列）
- 本年所有 daily 的 8 习惯打卡（全年统计 + 4 季度趋势）
- 本年完成 tasks（[tasks-query.md](tasks-query.md) Q3/Q5/Q6/Q7）+ 按季节分布
- **本年跨 repo git 提交**（Q10，按月/季度聚合数字 + 关键 commit 摘要）
- 项目维度分类（完成/进行中/暂停/放弃）
- 上一年 yearly 用于年初 vs 年底对比

Plan 段额外数据源：
- 上一年 yearly 全文（theme / vision / 正文）
- 上一年 Q4 quarterly 的「下季度改进方向」「commit to improve」
- 上一年 12 个月 theme 序列

---

## 目标文件结构

`<VAULT>/1-plan/5-yearly/YYYY.md`

### Frontmatter 字段切分

| 字段 | 类型 | 段 | 说明 |
|---|---|---|---|
| `theme` | string | **Plan** | 年度主题（如「破壁之年」「健康为先」） |
| `vision` | string | **Plan** | 年度愿景（一段话，描绘年底想看到的自己） |

模板没有显式 Review 段 frontmatter — 年度 review 的产出主要在正文 callout 和段落叙述里（见下方）。

### 正文

模板没有规定 section，正文几乎空白。可以让用户自由发挥，但 skill 应该**建议一个轻量结构**让用户填：

```
## 📜 上一年启示（来自上一年 yearly + 4 个 quarterly）
> 简短回顾上一年的轨迹

## 🌅 今年我想成为的样子（vision 展开）

## 🎯 年度三大支柱（focus areas）
1.
2.
3.

## 📍 年度关键指标
| 维度 | 起点 | 目标 | 衡量 |
|---|---|---|---|

## 🚀 标志性项目（年度里程碑）
- [ ]
- [ ]

## ⚠️ 年度避免清单
（这一年我承诺不再做的事）
-

## 📅 中期回顾节点
- Q1 末：
- Q2 末（年中）：
- Q3 末：
```

skill 建议这个结构（写入前问用户认不认）。也可以完全跳过结构，纯由用户口述、skill 整理。

## 路由：回顾还是规划？

- 12 月最后 2 周 → 默认年度回顾（写本年的）
- 1 月前 2 周 → 默认年度规划（写下年的）
- 中间 → 问用户

## 年度回顾流程

### 第 1 步：聚合数据

**1.1 读本年 4 个 quarterly** 的：
- 全部 frontmatter
- 「💡 季度关键洞察」4 子段
- 「📈 季度对比分析」

**1.2 读本年 12 个 monthly** 的 frontmatter：theme 序列

**1.3 全年习惯打卡**（约 365 个 daily）：用 obsidian-cli 算每个习惯的「全年完成天数 / 365」「四季度趋势」

**1.4 全年任务完成**（见 [tasks-query.md](tasks-query.md)）：
- **Q3 本年完成任务**：`✅ 2026-`，全 vault 范围（约几千条规模，**只看汇总数字 + 分组**，不读原文）
- **Q5 按 area 分组**：work / learning / health / family / technique 全年累计
- **按季节分布**：把本年完成任务按 Q1/Q2/Q3/Q4 拆开，看 4 个季度节奏
- **Q6 项目维度**：每个 `2-task_management/1-projects/` 项目全年完成 task 数；用这个数据辅助 1.5 的"完成/进行中/暂停/放弃"分类
- **Q7 年度积压**：跨年仍未完成的任务（截止日期早于今年），这是真正应该考虑放弃的

**1.5 项目维度**：列出本年活跃过的所有 `<VAULT>/2-task_management/1-projects/` 文件，分类「完成 / 进行中 / 暂停 / 放弃」

**1.6 阅读/学习**：从 `<VAULT>/4-knowledge_hub/Clippings/` 和 `Readwise/` 提取本年新增数量（粗略，用 file ctime 过滤）

**1.7 上一年 yearly** 的 `theme` 和 `vision` — 用于对比"年初的我 vs 年底的我"

### 第 2 步：组织展示

年度展示**讲故事**，不堆数据。3-4 张精挑的快照：

**展示 A：年度叙事线（12 个月主题串联）**
```
本年 12 个月的主题：
M1: 紧张/破壁  M2: 突破  M3: 收尾
M4: 调整      M5: 学习  M6: 稳定
M7: ...       ...

→ 看起来本年的 macro narrative 是：{LLM 总结一句话，等用户确认或修正}
```

**展示 B：年度对比 — 年初 vs 年底**
```
年初你写的 theme: "{X}"
年初你写的 vision:
"""
{原文}
"""

→ 一年过去了，回看这段 vision，你觉得：
  - 哪些已经成真？
  - 哪些还差一点？
  - 哪些其实方向变了？
```
（这是关键问题，**直接给用户看年初自己的话**，让他自己对比）

**展示 C：4 个季度 Top 3 收获大合集**
列出 4 × 3 = 12 条 learnings，让用户挑出**年度 Top 5**（比季度的 Top 3 更精）

**展示 D：习惯一年大表**
```
习惯       Q1     Q2     Q3     Q4     全年
🧘‍♂️ 冥想   65%    73%    70%    80%    72%   ↑稳步提升
💪 运动    72%    48%    51%    60%    58%   ⚠️Q2 后下滑未恢复
...
```

**展示 E：项目维度全景**
```
✅ 完成：{N 个项目}
🔄 进行中：{M 个}
⏸️ 暂停：{K 个}
❌ 放弃：{L 个}

完成的关键项：
- {项目 1}
- {项目 2}
...
```

**展示 F：观察到的年度模式**
```
🔍 我注意到（基于 12 个月数据）：
- 全年最高效的月份是 {X}，最低效是 {Y}
- 你在 weekly 反复出现的关键词：{N 个高频词}
- 4 个季度 commit to improve 里有 {Z} 重复出现 N 次 — 这事看起来一直没真正解决
- 年初 vision 里写的 {A B C}：A 成真，B 部分，C 完全没动
```

### 第 3 步：苏格拉底式深度对话

年度对话**深而不广**。建议 4-6 个问题，**对话式分次问**，每个问题用户回答后用他的话整理。

候选问题（择 3-5 个）：

- **回看年初**：
  - "年初你写 vision 时，最不敢承认能做到的是哪一条？现在回看？"
  - "年初你担心但没写出来的事情，过去一年发生了吗？"

- **关于身份**：
  - "今年的你比去年的你，最大的不同是什么？（不是做了什么，而是变成了什么样的人）"
  - "如果一个一年前的朋友今天遇到你，他会注意到你身上的什么变化？"

- **关于代价**：
  - "今年你拿下了 {X}，但代价是什么？（关注没说出口的取舍）"
  - "今年放弃了 {Y 项目}。回看是对的决定吗？"

- **关于反复**：
  - "{Z 主题} 四个季度都出现了但都没真正解决。它对你重要吗？还是只是一个执念？"

- **关于未来锚点**：
  - "如果只能挑一个 word 概括今年，是什么？"（→ 后面会变成下一年的对照）

**避免**：
- 不要按部就班问完所有问题
- 不要急着写入，让对话有沉淀的时间
- 用户给出短答案时，**不要满足**，问"为什么"

### 第 4 步：组织成文 + 确认 + 写入

年度回顾的产出可以比较**长**（vision/narrative 段可能 200-500 字），但应该**用户自己的语言**。

写入策略：

**4.1 Frontmatter**（最后再写）
- `theme`：一句话主题（用户拍板）
- `vision`：可以延用年初的（如果还成立），也可以加一段「年底校准」备注

**4.2 正文**（如果用户认可第 1 步建议的轻量结构）
按建议结构 Edit 写入。如果用户不要结构，就让用户口述、skill 整理成段落 → Edit 一段段写。

不必所有 section 都填满，留空也行。

**4.3 备份**：年度文件值得**额外**保存一份对话原文到 `notes/`，方便日后回看（可选，问用户）。

## 年度规划流程

### 第 1 步：聚合上一年数据

**1.1 上一年 yearly** 全文（`theme` / `vision` / 正文）
**1.2 上一年 Q4** 的「下季度改进方向」「commit to improve」「Top 3 learnings」
**1.3 上一年 12 个月主题序列**

### 第 2 步：组织展示

```
📜 上一年（{YYYY-1}）总结：
- Theme: {X}
- 你写的 vision（去年初）:
  """
  {原文}
  """
- 年度 Top 3 学到的：{1, 2, 3}（来自 Q4 review）
- 上一年最后承诺改进的：{Y}

→ 站在这一年起点，几个候选 theme：
  1. {基于上年遗留}
  2. {基于上年突破}
  3. {基于用户暗示的方向}
```

### 第 3 步：苏格拉底式追问 3-5 个

- "如果这一年只能聚焦一个 word/phrase，你想要是什么？"
- "上一年想做但没做的 {X}，这一年还要做吗？为什么这次会不一样？"
- "上一年最让你满意的事情是 {Y}，这一年想复制这种状态还是探索新方向？"
- "12 个月后，你希望别人怎么描述这一年的你？"
- "这一年最大的风险/不确定性是什么？想为它做什么准备？"

### 第 4 步：写入

**4.1 Frontmatter**
- `theme`: 用户拍板
- `vision`: 一段话（≤ 200 字），用户口述 + skill 整理

**4.2 正文**
建议结构同上（年度回顾的 section 结构）。先只填规划性的几段：
- 年度三大支柱
- 年度关键指标
- 标志性项目
- 年度避免清单
- 中期回顾节点

---

## 节奏建议

- **年度回顾**：建议分 2-3 次跑（避免一次性 1 小时太累）
  - 第 1 次：数据聚合 + 年初 vs 年底对比 → 留 1-2 天沉淀
  - 第 2 次：身份/代价/反复 三类深问 → 整理成段
  - 第 3 次：theme/vision 校准 + 备份
- **年度规划**：可以 1 次跑完（45 min），但 theme 和 vision 建议留 24h 沉淀再定

skill 不主动催，看用户节奏。
